# 넘버 트럭 러너 (Number Truck Runner)

고철 손수레가 산술 게이트로 전력(숫자)을 불리며 도로 끝까지 질주하는, 한손 조작 모바일 캐주얼 러너 프로토타입입니다.
숫자가 임계치를 넘을 때마다 트럭의 외형이 극적으로 진화하며, 손수레에서 우주적 존재로 이어지는 성장 서사를 완성합니다.

자세한 설계 의도와 밸런스 튜닝 과정은 [DESIGN.md](./DESIGN.md)에 정리되어 있습니다.

## 실행 방법

```bash
npm install
npm run dev       # 개발 서버 (http://localhost:5173)
npm run build     # 정적 빌드 (dist/)
npm run preview   # 빌드 결과 미리보기
npm run typecheck # 타입체크만
npm run sim       # 밸런스 시뮬레이션 (헤드리스, 콘솔 리포트)
```

## 조작법

- **모바일**: 화면을 좌우로 드래그(또는 원하는 위치를 탭)하면 트럭이 해당 레인으로 이동합니다.
- **데스크톱**: 마우스 드래그, 또는 방향키(←/→) / A·D 키로 레인 전환.
- 트럭은 자동으로 전진하므로, 플레이어는 **레인 선택**에만 집중하면 됩니다.

## 게임 흐름

1. 시작 화면 → 스테이지 인트로(신규 메커닉 소개) → 플레이
2. 3레인 갈림길에서 산술 게이트(`+N%` / `−N%` / `×N` / `÷N`) 중 하나를 선택해 전력을 불림
3. 전력이 임계치를 넘으면 트럭이 즉시 진화(화면 슬로우모 + 파티클 + 티어 카드 연출)
4. 바리케이드 · 적 차량 · 톨게이트 · 좁은 통로 · 보너스 레인(고위험 고보상) 등을 레인 선택으로 돌파
5. 스테이지 끝의 보스(3페이즈 돌파형)를 처치하면 클리어, 7개 스테이지를 모두 통과하면 엔딩
6. 전력이 바닥나면 게임 오버 → 즉시 재도전 가능

콘솔에서 `window.__debug.invincible(true)`, `window.__debug.speed(2)`, `window.__debug.power(1e20)` 로 무적/배속/전력 강제 설정이 가능합니다.

## 코드 구조

```
src/
  core/        게임 루프(engine.ts), 입력 처리, 숫자 포맷(K/M/B/T/aa…), RNG
  data/        전부 상수/데이터 — 여기 숫자만 바꾸면 밸런스가 바뀝니다
    tiers.ts     진화 티어 래더 (임계치, 이름, 로어, 외형 파라미터)
    stages.ts    7개 스테이지 구성 (길이, 참조 전력, 세그먼트 가중치, 바이옴)
    gates.ts     스테이지별 산술 게이트 값 분포
    enemies.ts   적/톨게이트/보너스레인/보스 강도 설정
    balance.ts   전역 물리/연출 상수
  entities/    트럭, 트랙 세그먼트, 파티클 데이터 구조
  systems/     스포너(절차적 스테이지 생성), 충돌·전투 해결, 진화 판정, 카메라·연출(juice)
  render/      캔버스 드로잉 (도로, 트럭, 게이트/장애물, 이펙트, HUD)
  ui/          DOM 오버레이 (시작/인트로/클리어/게임오버/엔딩/일시정지)
balance-sim/   헤드리스 밸런스 시뮬레이션 (빌드와 독립적으로 tsx로 실행)
```

## 밸런스 튜닝 가이드 (숫자만 바꿔서 조정하기)

- **성장 속도를 늦추고 싶다면**: `data/gates.ts`의 각 스테이지 `opWeights.mul`(곱셈 등장 확률)이나 `mulRange`(배수 크기)를 낮추세요.
- **특정 스테이지가 너무 쉽거나/어렵다면**: `data/enemies.ts`의 `BOSS_CONFIGS[stageIdx].phaseFactors` (보스 3페이즈 요구 전력의 배율)를 조정하세요. 값이 클수록 어려워집니다.
- **적/톨게이트 강도**는 `data/enemies.ts`의 `ENEMY_CONFIGS` / `TOLLGATE_CONFIGS`가 그 시점의 "참조 전력"(`data/stages.ts`의 `startReferencePower`/`endReferencePower`) 대비 배율로 정의됩니다. 참조 전력을 실제 도달 전력과 어긋나게 바꾸면 난이도가 왜곡되니, 반드시 `npm run sim`으로 재검증하세요.
- **진화 페이싱**을 바꾸려면 `data/tiers.ts`의 `threshold` 값들을 조정하세요.
- **스테이지 등장 빈도**(바리케이드/적/톨게이트/보너스레인 비율)는 `data/stages.ts`의 `segmentWeights`에서 조정합니다.
- 값을 바꾼 뒤에는 항상 `npm run sim`을 실행해 스테이지별 클리어율·중앙값 전력·사망 지점을 확인하세요.

## 설계 노트 (요약)

- **왜 3레인 + 3지선다 게이트인가**: 모바일 세로 화면에서 한눈에 읽히면서도, 매번 "무엇을 고를지" 판단할 거리를 준다.
- **왜 진화가 승급/강등을 하나의 규칙으로 처리하는가**: `tier = 현재 전력 이상을 만족하는 가장 높은 임계치`라는 단일 규칙이면 별도 분기 없이 항상 일관된 상태를 보장한다.
- **왜 `+`/`−` 게이트가 고정값이 아니라 퍼센트인가**: 밸런스 시뮬레이션 결과, 고정값은 후반부에 완전히 무의미해져 "안전 위주" 플레이가 조기에 전멸하는 문제가 있었다. 퍼센트 기반으로 바꿔 정책 간 격차를 합리적인 수준으로 좁혔다 (자세한 경위는 DESIGN.md §9 참고).
- **왜 티어가 20단계(0~19)까지 있는가**: 곱셈 게이트가 여러 스테이지·세그먼트에 걸쳐 누적되면 전력이 자연스럽게 10³⁷ 수준까지 도달한다는 것이 시뮬레이션으로 확인되어, 초반 설계(10단계, 3×10¹⁰ 상한)로는 스테이지 4~7 내내 트럭이 같은 모습으로 남아있었다. 진화 서사가 게임 끝까지 계속되도록 래더를 확장했다.
